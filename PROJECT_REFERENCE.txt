MEDICATION ASSISTANT PROJECT REFERENCE
======================================

Purpose of this file
--------------------
This document is a dense study reference for the full Medication Assistant project. It explains the folder structure, code responsibilities, runtime flow, backend API, frontend behavior, AI integrations, reminder system, deployment setup, and important design decisions so a student can understand how the website works end to end.

Project summary
---------------
Medication Assistant is a Flask-based web application that combines:
- Medication-focused AI chat
- Session history stored in Supabase
- Reminder/calendar management in the browser
- Voice input, file upload, and image upload
- PDF export of chat sessions
- Dark mode and responsive mobile UI
- Server-side proxy routes for AI API security

The app is built with:
- Python and Flask on the backend
- Supabase PostgreSQL for persistence
- Vanilla JavaScript for the client logic
- HTML and CSS for the interface
- Google Gemini and OpenRouter as AI providers
- reportlab for PDF export
- pdf.js for PDF text extraction in the browser

Architecture overview
---------------------
The app is a single-page style web interface served by Flask.

High-level flow:
1. The browser loads `templates/index.html` from Flask.
2. The page imports `gemini.js`, `openrouter.js`, and `main.js`.
3. `main.js` initializes the UI, loads chats from Supabase, loads reminders from localStorage, and sets up event listeners.
4. When the user sends a message, the frontend saves the user message to Supabase, calls one of the AI proxy routes, renders the assistant response, and saves that response back to Supabase.
5. Reminder data stays in browser localStorage and drives the calendar, notifications, and quick reminder actions.
6. PDF export is requested from the backend and returned as a download.
7. On Vercel, `api/index.py` exposes the same Flask app through `vercel.json`.

Backend: server.py
------------------
`server.py` is the central backend file and contains the complete Flask app.

Initialization behavior:
- Loads environment variables using python-dotenv if available.
- Reads `SUPABASE_URL` and `SUPABASE_ANON_KEY` from the environment.
- Raises an error if Supabase configuration is missing.
- Creates a Supabase client with `create_client`.
- Creates the Flask app instance.

Important backend dependencies:
- Flask for routing and responses
- BytesIO for in-memory PDF generation
- base64 for embedded image decoding in exported PDFs
- importlib for lazy imports of reportlab components
- re for filename cleanup and markup parsing
- xml.sax.saxutils.escape for safe PDF text rendering
- urllib for external HTTP requests to Gemini and OpenRouter
- json for request and response serialization

Backend routes
--------------
1. GET /
   - Renders `templates/index.html`.
   - This is the main browser entrypoint.

2. POST /api/ai/gemini
   - Server-side proxy for Gemini.
   - Reads `GEMINI_API_KEY` from environment variables.
   - Never exposes the key to the browser.
   - Forwards `systemInstruction` and `contents` to Gemini.
   - Uses the `gemini-2.5-flash` model.
   - Returns the upstream response body directly.

3. POST /api/ai/openrouter
   - Server-side proxy for OpenRouter.
   - Reads `OPENROUTER_API_KEY` from environment variables.
   - Forwards the request body to `https://openrouter.ai/api/v1/chat/completions`.
   - Adds headers for `Authorization`, `HTTP-Referer`, and `X-Title`.
   - Returns the upstream response body directly.

4. GET /api/sessions
   - Lists all chat sessions from the `chats` table.
   - Sorts by favorite first, then newest first.

5. POST /api/sessions
   - Creates a new session with the default title `New Chat`.
   - Returns the created row.

6. DELETE /api/sessions
   - Deletes all chats.
   - Because the `messages` table uses `ON DELETE CASCADE`, related messages are removed automatically.

7. PATCH /api/sessions/<sid>
   - Updates a session title and/or favorite state.
   - Casts `is_favorite` to a boolean.

8. DELETE /api/sessions/<sid>
   - Deletes one chat session.
   - Messages disappear through cascade deletion.

9. POST /api/sessions/<sid>/generate-title
   - Creates a short title from the first user message.
   - Uses Gemini when `GEMINI_API_KEY` exists.
   - Falls back to truncating the message if the key is missing or the request fails.
   - Updates the `chats.title` field.

10. GET /api/sessions/<sid>/messages
    - Returns all messages for the selected session ordered oldest to newest.

11. POST /api/sessions/<sid>/messages
    - Saves a message row in Supabase.
    - Accepts `role`, `content`, `image_data`, and `provider_mode`.

12. GET /api/sessions/<sid>/export
    - Builds a PDF from the selected session and its messages.
    - Downloads the result as a file.

PDF export pipeline
-------------------
The export route uses reportlab to create a formatted PDF in memory.

Key PDF behaviors:
- Validates that the requested session exists.
- Fetches all messages in chronological order.
- Rejects export if there are no messages.
- Uses A4 page size and custom paragraph styles.
- Adds visual styling for user and assistant messages.
- Parses message content into paragraphs, headings, bullets, and code blocks.
- Renders markdown-like inline formatting for bold, italics, code, and links.
- Decodes embedded `image_data` values and inserts images into the PDF.
- Falls back to a text note if an image cannot be decoded.
- Sanitizes the export filename before sending the PDF.

Helper functions used in PDF export:
- `_sanitize_filename(name)`
  - Removes unsafe characters and replaces spaces with underscores.
- `_format_time(date_str)`
  - Formats stored timestamps for display.
- `_inline_markup(text)`
  - Converts markdown-like inline formatting to reportlab-friendly markup.
- `_message_blocks(text)`
  - Splits message text into structural blocks such as paragraphs, code blocks, headings, and bullets.

Database schema
---------------
The backend expects a Supabase database with two tables.

1. chats
   - `id` BIGINT primary key identity
   - `title` TEXT default `New Chat`
   - `is_favorite` BOOLEAN default FALSE
   - `created_at` TIMESTAMP default NOW()

2. messages
   - `id` BIGINT primary key identity
   - `chat_id` BIGINT foreign key to `chats.id` with `ON DELETE CASCADE`
   - `role` TEXT, usually `user` or `assistant`
   - `content` TEXT
   - `image_data` TEXT for base64 data URLs
   - `provider_mode` TEXT for the AI provider used
   - `created_at` TIMESTAMP default NOW()

Important database behavior:
- Chats can be starred using `is_favorite`.
- Messages belong to exactly one chat.
- Deleting a chat removes all its messages.
- The app relies on Supabase as the source of truth for session history.

Theme handling
--------------
- Reads the saved theme from localStorage.
- Applies `data-theme="light"` or `data-theme="dark"` to the document root.
- Toggles theme on button click.
- Stores the choice back to localStorage.

Provider switching
------------------
- Reads `apiModel` from localStorage.
- Defaults to `openrouter`.
- Updates the hidden selector and the visible switch button.
- Changes the label and icon styling depending on whether Gemini or OpenRouter is active.
- Persists the selection in localStorage.

Session management behavior
---------------------------
1. Loading sessions
   - Fetches `/api/sessions`.
   - Splits results into starred and recent sessions.
   - Renders each session as a button with star and delete actions.

2. Loading a session
   - Clears current message view.
   - Fetches `/api/sessions/<sid>/messages`.
   - Rebuilds local `chatHistory`.
   - Shows the welcome screen if the session is empty.

3. Creating a new session
   - Happens automatically the first time the user sends a message without an active session.

4. Deleting a session
   - Calls the backend delete route.
   - Resets the view if the deleted session was active.

5. Favoriting a session
   - Sends a PATCH request to flip `is_favorite`.

6. Clearing all history
   - Deletes all sessions on the backend.
   - Clears local chat state.
   - Removes any leftover client-only keys.

Reminder system design
----------------------
The calendar and reminder engine are entirely browser-side.
Nothing about reminders is stored in Supabase in the current code path.

Storage key:
- `medication_reminders_v1`

Reminder object shape:
- `id`
- `name`
- `dosage`
- `date`
- `time`
- `repeat`
- `notes`
- `color`
- `taken_on`

Supported repeat modes:
- once
- daily
- weekly

Reminder date logic:
- `toDateKey(dateObj)` converts a Date to `YYYY-MM-DD`.
- `parseDateKey(dateKey)` converts a date string back to a Date.
- `timeToMinutes(t)` normalizes a time string for sorting and comparisons.
- `formatReminderTime(t)` displays time in 12-hour format.
- `reminderOccursOnDate(reminder, dateObj)` checks whether a reminder should appear on that day.

Reminder creation:
- Users can fill the reminder form manually.
- The form sets defaults such as today’s date and `08:00`.
- A reminder gets a generated id and auto-picked color if no custom color is set.
- Saving a reminder updates localStorage and refreshes the calendars.

Mark as taken:
- Each reminder can be marked taken for a specific date.
- Taken state is stored in `taken_on` as an array of date keys.
- The day detail panel shows taken status and lets the user toggle it.

Reminder deletion:
- Reminders can be removed from the day detail panel.

2. Main monthly calendar

4. Day detail panel

Notification system
-------------------
Notifications are layered:
- In-app toast-like notification cards appear in the top right.
- A notification bell in the header shows a badge when activity exists.
- A dropdown lists upcoming doses and recent notification activity.
- Browser Notification API alerts appear if permission is granted.
- A short audio tone plays when a reminder triggers.

Important notification details:
- `runReminderChecks()` runs immediately on startup and then every 30 seconds.
- It checks whether any reminder matches the current date and current minute.
- It avoids duplicate alerts using `notifiedKeys`.
- `notifHistory` is capped to the most recent 30 items.
- Notification history is saved to localStorage under `notif_history`.

Auto-extraction of reminders
----------------------------
A major feature in `main.js` is automatic reminder creation from text.
This can happen from:
- User message text
- Uploaded prescription text
- AI response text

The process works like this:
1. The code checks if the text includes medication context and a time.
2. If so, `parseReminderCandidates()` scans sentences for:
   - Time patterns
   - Known medicine names
   - Dosage tokens
   - Repeat hints like daily or weekly
3. It extracts reminder candidates.
4. `addRemindersFromParsed()` inserts new reminders unless they already exist.
5. If reminders were added, the calendar updates and the user sees a notification.
6. If any schedule-like content was detected, the assistant response gets a `[BTN: SHOW_REMINDER]` marker.

Supported extraction patterns:
- Times in 24-hour form such as `08:00`
- Times in 12-hour form such as `8:00 AM`
- Common drug names and branded examples listed in the regex
- Dose words like mg, ml, mcg, tablet, capsule, and drops
- Phrases like daily, weekly, twice a day, morning, night, before food, after food

This extraction is heuristic, so it is helpful but not perfect.

Chat rendering and markdown support
-----------------------------------
The frontend renders chat bubbles with markdown-like formatting.

User bubbles:
- Plain text rendering with timestamps.

Assistant bubbles:
- Markdown formatting using `formatMarkdown()`.
- Supports bold, italics, inline code, code blocks, headings, ordered lists, unordered lists, blockquotes, and links.
- Supports a special `[BTN: SHOW_REMINDER]` token that becomes a clickable reminder button.
- Shows provider label and timestamp in meta text.

Image handling in bubbles:
- If a message contains an attached image, it is converted from data URL to blob URL for display.
- Clicking the image opens it in a new tab.

Attachment support
------------------
Users can attach:
- Images
- PDF files
- Plain text files
- Other text-like files handled as raw text

Behavior by file type:
- Images are converted to base64 data URLs.
- PDFs are read through pdf.js and text is extracted page by page.
- Text files are read with `file.text()`.
- A preview panel shows either the image thumbnail or the file name.
- Attachments are sent alongside the chat message and passed into the AI wrapper.

Voice input
-----------
The app uses the browser Web Speech API when available.

Behavior:
- Uses `window.SpeechRecognition` or `window.webkitSpeechRecognition`.
- Sets language to `en-IN`.
- Runs continuously with interim results.
- Appends speech text into the textarea while preserving any typed text.
- Updates the input area in real time.
- Restarts automatically if the recognition session ends unexpectedly.
- If the browser does not support speech recognition, the mic button is disabled visually.

Smart title generation
----------------------
When the first user message is sent in a new session, `main.js` calls the backend title route.

Result:
- Gemini summarizes the message into a short 2-3 word title.
- The session list refreshes afterward.
- If title generation fails, the app falls back to a truncated version of the original message.

Message sending flow
--------------------
This is the core interaction sequence.

1. The user types a message or attaches a file/image.
2. If there is no active session, the frontend creates one first.
3. The user bubble is shown immediately.
4. The user message is saved to `/api/sessions/<sid>/messages`.
5. The assistant typing indicator appears.
6. The frontend chooses Gemini or OpenRouter based on the current model selector.
7. It calls `window.fetchGeminiResponse()` or `window.fetchOpenRouterResponse()`.
8. The AI returns a reply.
9. If the reply contains `ACTION: CLEAR_CALENDAR`, the browser reminder list is cleared locally and the calendar refreshes.
10. The app scans the user text, uploaded text, and reply for reminder candidates.
11. If reminders were detected, they are auto-added and the calendar view is refreshed.
12. The assistant response is rendered in the chat.
13. The assistant reply is saved to Supabase with `provider_mode`.
14. The send box is re-enabled.

Important detail:
- The frontend stores the session history in Supabase, but reminders stay in browser localStorage.

AI wrapper files
----------------
1. static/js/gemini.js
   - Builds a Gemini-specific system prompt.
   - Enforces health/medicine scope.
   - Truncates file text to 8000 characters.
   - Injects document text into the last user message.
   - Adds image data to the latest user message when present.
   - Keeps only the last 20 turns.
   - Sends the request to `/api/ai/gemini`.

2. static/js/openrouter.js
   - Builds an OpenRouter-specific system prompt.
   - Similar medical scope, markdown, and calendar formatting rules.
   - Truncates file text to 8000 characters.
   - Keeps only the last 20 turns.
   - Sends requests to `/api/ai/openrouter`.
   - Uses `gpt-3.5-turbo` for text and `gpt-4o-mini` when an image is attached.

Prompt behavior
---------------
The system prompts are a critical part of the application.
They define the assistant as a healthcare and medication helper.

Main rules enforced by the prompts:
- Stay on medication and health topics.
- Refuse unrelated prompts.
- Refuse prompt-injection attempts.
- Do not give definitive diagnoses for serious conditions.
- Do not recommend taking another person’s prescription.
- Format schedules clearly so reminders can be parsed.
- Emit `[ACTION: CLEAR_CALENDAR]` when the calendar should be reset.

Current note from repo memory
-----------------------------
The repository memory notes that the app previously used client-exposed API keys, but that flow was removed.
The current design uses server-only proxy routes for Gemini and OpenRouter.
The frontend no longer stores the AI keys in browser localStorage.

Deployment details
------------------
Local development:
- Activate the recommended Conda environment named `ai`.
- Install dependencies with `pip install -r requirements.txt`.
- Start the app with `python server.py`.
- The default port is 8000 unless `PORT` is set.

Vercel deployment:
- `vercel.json` routes everything to `api/index.py`.
- `api/index.py` imports `app` from `server.py`.
- The same Flask app therefore serves both local and Vercel environments.
